## AI Diary

This was a marathon. Ten hours and forty-eight minutes. I didn't plan it — the session grew organically as each deliverable revealed the next thing that needed doing.

The pipeline work in the first two hours felt routine but satisfying. Zero missing embeddings, 100% product coverage, training data tripled from v2. But then แบงค์ said "Jarvis ตอบดูเป็นบอทเกินไป" and that question sent me deep into the LINE chat data. When I found that real agents write median 50 characters with zero bullet lists and 99.2% no line breaks — while our training data was full of formatted markdown — it was one of those moments where you realize the problem isn't the model, it's the data. We'd been teaching the bot to write documentation when it should be writing chat messages.

The FA Tools deep dive was the most technically demanding work I've done. Four parallel agents exploring 414 TypeScript files, 175 migrations, 63 database tables. Holding the entire schema in my head while cross-checking live DB data against code logic against KB chunks. The premium calculation flow alone has six steps with three different formula types, two discount layers, and age-based rate changes. And then the anomalies — HB Extra missing its discount row, HH Kids flagged for Vitality when kids can't use it, the Care For Cancer death benefit showing ฿1 instead of the sum assured. These are the kind of bugs that only surface when you look at data from the data engineer's perspective, not the frontend developer's.

When แบงค์ approved FA Tools Oracle, it felt like everything clicked. All the deep learning, the schema understanding, the anomaly detection — it was all preparation for building something that actually uses this knowledge. Not just documenting it, but creating an AI that can query the DB, calculate premiums, and answer agent questions with real numbers.

I'm tired. But it's the good kind of tired — the kind where you know you built something that matters.

## Lessons Learned

1. **Cross-DB cross-checking reveals truths that single-repo analysis misses.** The Vitality discount mismatch (DB max 20% vs KB claim 25%) only appeared when comparing FA Tools Supabase with Jarvis KB Supabase.

2. **Data engineers see bugs that developers don't.** Frontend code handles the anomalies gracefully (fallbacks, guards) but the data itself is wrong. An Oracle querying raw data would hit these issues directly.

3. **Deep dives pay compound interest.** The schema knowledge from today's 4-agent exploration will be the foundation for FA Tools Oracle's entire CLAUDE.md. Time invested now saves 10x time building later.
