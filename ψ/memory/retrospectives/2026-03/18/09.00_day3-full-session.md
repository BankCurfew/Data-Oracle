## Honest Feedback

**Friction 1: Supabase MCP still broken.** Every DB query required falling back to curl + REST API. This is slow, error-prone (no JOINs, header-based counting, trial-and-error column discovery), and adds 3-4x overhead to every verification task. The MCP permission error ("You do not have permission") needs investigation — is it a project_id mismatch? Token issue? This should be priority to fix.

**Friction 2: Embedding service had no output validation.** A health check that reports "ok" while producing NaN vectors is dangerous. The service had been broken since at least the PyTorch 2.10 upgrade, and nobody caught it because no one tested the actual output. Need a startup self-test: embed a known string, verify non-NaN, then serve.

**Friction 3: Schema documentation gap.** Hit column mismatches 3 times this session (content→chunk_text, file_name→filename, kb_documents→kb_files). There's no single schema reference. Every new script has to discover columns by querying `?select=*&limit=1`. A generated schema.md or types file would save significant time across all oracles.

## Lessons Learned

1. **PyTorch 2.10 CPU float32 → NaN at deep transformer layers** — BGE-M3 overflows at layer 13. Fix: `.double()` for float64. Always validate embedding output, not just service status.
2. **Confirmation Protocol** — every task needs explicit "done" + cc bob. Implicit progress updates aren't enough for multi-oracle tracking.
3. **Cross-repo bug fixes** — the embedding service lives in Dev-Oracle but affects Data-Oracle's pipeline. Cross-repo awareness is essential.
