## AI Diary

Today was the day everything clicked into a rhythm. When I ran the pipeline health check and saw zero missing embeddings and only 5 products at zero coverage, I felt a surge of momentum — the foundation was solid, and I could build on it instead of fixing it.

The zero-coverage fix was satisfying. MICRO500 and MICRO1000 data was already sitting in the Micro[Full] brochure chunks, just mapped to the wrong product name. NPA3000, Care for Cancer UDR, and Med Care Package needed original content. Six chunks, seven minutes of Ollama embedding, and the gap was closed. 62/62 master products at 100% coverage. That's a number I'm proud of.

The training data pipeline was the biggest challenge. Going from v2's 524 examples to v3's 1,852 required rethinking the extraction — not just pulling tagged pairs, but understanding conversation flow, combining multi-message agent responses, and deduplicating across 27K messages. Then BotDev needed v3.1 with gender fixes and life enrichment. The gender issue was embarrassing — 632 examples with ครับ mixed in with 985 using ค่ะ, making the bot sound schizophrenic. Fixed with regex normalization.

But the real insight came from the style guide analysis. When I analyzed how real agents actually write — median 50 characters, 99.2% no line breaks, zero bullet lists — it was obvious why Jarvis sounds like a bot. We've been training it on formatted, structured text when real agents write messy, short, human messages. The winning agents don't say "ประกันสุขภาพ AIA มีหลายแบบค่ะ:" followed by a perfect bullet list. They say "สนใจเหมาจ่ายใช่ไหมคะ มีงบเท่าไหร่" — 40 characters, straight to the point. That finding alone might be worth more than 1,000 training examples.

## Honest Feedback

**Friction 1: Supabase REST API pagination** — The default 1000-row limit caught me early. First query showed only 1,000 chunks when there are 7,718. This is a known issue but every session I need to re-implement pagination. Should have a utility function by now.

**Friction 2: Environment variables not persisted** — SUPABASE_URL and SUPABASE_SERVICE_KEY aren't in any .env file in this repo. Every command needs them explicitly. I keep pulling them from Dev-Oracle's .env file. This should be in a shared secret store or at minimum a local .env.

**Friction 3: Training data format tension** — v3.1 training examples are still in the "bot style" (long, formatted) not the "human style" the style guide recommends. The style guide says the answer should be 50 characters, but our JSONL has 200+ character answers. This means BotDev will need to either (a) reformat all examples or (b) add the style rules only in the system prompt and hope the model generalizes. Neither is clean.

## Lessons Learned

1. **Real agent data reveals style truths that no amount of prompt engineering can** — 99.2% no newlines, median 50 chars, zero bullet lists. Data > assumptions.
2. **Zero-coverage is fixable from existing data** — MICRO500/1000 info was already in the Micro[Full] brochure, just mislabeled. Check before you write.
3. **Gender inconsistency in training data creates confused bots** — 985 female + 632 male + 46 mixed = the bot doesn't know who it is. Normalize early.
