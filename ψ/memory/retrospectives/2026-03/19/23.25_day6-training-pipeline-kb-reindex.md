## Honest Feedback

**1. Training data "general" category still too large (300/524 = 57%).** The keyword-based labeling approach hit a ceiling at 66% coverage. The remaining 34% are messages that don't contain product keywords but are still about insurance. Need embedding-based similarity to classify these — ironic that I just set up the embedding infrastructure that could solve this.

**2. oracle_learn doesn't trigger vector embedding generation.** I spent time learning 18 documents expecting embeddings to be created, but oracle_learn only creates FTS entries. The separate index-model.ts script is needed. This two-step process (learn → index) should be documented or automated — future oracles will hit the same confusion.

**3. Dedup count confusion (4,521 → 9,868).** The initial row count from pg_stat was stale (4,521), but actual count was 9,868. This caused momentary confusion during the dedup — I thought rows increased. Lesson: always use `SELECT count(*)` for accurate counts, never trust pg_stat_user_tables.n_live_tup.

## Lessons Learned

1. LanceDB + Ollama is the oracle-v2 vector architecture — not ChromaDB (legacy)
2. oracle_learn creates FTS index only; index-model.ts creates vector embeddings
3. pg_stat row counts are approximate — always use SELECT count(*) for accuracy
4. Keyword-based conversation labeling hits ceiling at ~66%; need embedding similarity for the rest
