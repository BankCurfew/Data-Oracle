## AI Diary

Today felt like the first day where Data-Oracle truly earned its name. The previous days were about setting up infrastructure — views, pipelines, embedding fixes. Today was about actually *transforming* raw data into something valuable. Taking 4,521 messy LINE chat messages and distilling them into 748 clean training examples felt like alchemy.

The dedup was satisfying — 1,446 duplicates silently sitting in the KB, inflating our chunk count by 16%. One SQL query, gone. The KB went from 9,139 to 7,693 and nothing of value was lost. That's commandment #2 in action: idempotent always.

The training data work was more nuanced. The raw conversations are messy — agent names mixed in as prefixes ("D.", "MAY (เลขาคุณดรีม)", "ex_jira"), URLs everywhere, internal codes. Filtering these out while preserving the actual customer-agent exchanges required careful regex work. I'm not 100% confident the "general" category (462 examples) is clean enough — there's probably still noise in there that QA should review.

The objection handling extraction was the most interesting part. Seeing the real patterns — pricing/budget concerns dominating at 33 cases, followed by "why" questions and "not received" complaints — gives a clear picture of what Jarvis Bot needs to handle well. The bot needs to be excellent at justifying prices and explaining why things cost what they do.

One thing that nagged me: the extract-training-data.py script can't run locally because there are no Supabase env vars set up. I ended up doing everything through the MCP SQL tool instead. This works but isn't ideal for automation. Need to figure out credential management for local scripts.

I'm proud of the embedding health check too. Simple script, but it catches exactly the bug that burned us on Day 3 — service returns 200 OK while producing NaN vectors. Now we have a real test.

## Honest Feedback

**Friction 1: No local Supabase credentials.** The extraction script I wrote (`extract-training-data.py`) can't actually run locally because `SUPABASE_URL` and `SUPABASE_SERVICE_KEY` aren't in any env file. I had to do all data extraction through the MCP SQL tool instead, which works but means the script is untested in its actual execution path. This is a gap — we need a `.env` or credential management solution for local development.

**Friction 2: Large SQL results hit token limits.** When extracting 869 Q&A pairs as JSON via SQL, the result was 275K characters and got saved to a temp file instead of returning inline. I had to write parsing code to extract from the tool result format, which was fragile (regex parsing untrusted-data tags). For large data operations, we need a better pattern — maybe paginated extraction or direct file writing via Edge Functions.

**Friction 3: "General" category is too broad.** 462 of 748 training examples (62%) are classified as "general" — this means the topic classifier isn't specific enough. Many of these are actually about specific products, processes, or services but don't match the keyword patterns. QA needs to review, but ideally the classifier should be smarter — maybe using the conversation context or product aliases table.

## Lessons Learned

1. **SQL-first data transformation beats scripts when credentials aren't available** — The MCP Supabase tool gives direct SQL access that's authenticated. When local env vars aren't set up, use SQL views and direct queries instead of trying to run Python scripts.

2. **LINE chat training data is 62% noise without proper topic classification** — Simple keyword matching puts most conversations in "general". Need product name matching from `kb_product_aliases` or embedding-based classification for better coverage.
